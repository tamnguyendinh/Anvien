# Contributing to Anvien

How to work on the current Anvien codebase, run the right checks, and open pull requests without changing the local-first runtime contract.

## Project Model

Anvien is a local code-intelligence tool:

- `anvien analyze` indexes a local repository into `<repo>/.anvien/`.
- `anvien mcp` exposes indexed repositories to MCP clients over stdio.
- `anvien serve` exposes the same local runtime to the Web UI over `127.0.0.1`.
- `anvien-web/` is a browser client. It does not own indexing, graph storage, or runtime locks.
- `anvien-launcher/` is an optional Windows convenience launcher around the same backend and Web UI.
- Chat runs through the local session bridge and provider adapters. Keep provider state scoped to sessions and explicit repo context.

Keep these invariants intact:

- Do not add an Anvien-hosted cloud service to the runtime path.
- Do not require API keys to be stored in Anvien.
- Do not make the launcher mandatory. `anvien serve` must remain the direct backend entry point.
- Do not move indexed repositories into a managed workspace just to make repo switching work.
- Repo context should be explicit by repo name/path/session. Avoid new process-global "active repo" state for graph reads.

## Repository Layout

| Path | Purpose |
|------|---------|
| `cmd/`, `internal/` | Go CLI, MCP server, local HTTP API, ingestion pipeline, LadybugDB access, embeddings, contracts, AI context generation, and session bridge. |
| `anvien/` | npm package metadata, package lifecycle scripts, and generated production runtime binary under `anvien/bin/`. |
| `anvien-web/` | Vite + React Web UI for graph exploration, repo picker/analyze flows, runtime diagnostics, and session chat. |
| `anvien-launcher/` | Windows launcher, protocol handler, server wrapper, bundled web/backend assets. |
| `docs/` | Plans, design notes, evidence, benchmarks, and investigation history. |
| `.github/` | CI, release, PR, and repository automation. |

## Generated And Packaged Artifacts

Do not edit generated outputs as the permanent source of truth:

- `AGENTS.md`, `CLAUDE.md`, and `.claude/skills/anvien/**` are generated from the registry in `internal/aicontext/aicontext.go` and embedded skill Markdown under `internal/aicontext/skills/*.md`.
- The generated Anvien skill set is task-facing only. Do not reintroduce a generated `anvien-ai-context` skill or any generated skill that instructs agents to modify its own generator.
- Web contracts come from Go contracts/generators, including `internal/contracts/` and `cmd/generate-web-contracts/`.
- `anvien/bin/anvien.exe` is the packaged production CLI/runtime generated by package or launcher builds.
- `anvien-launcher/server-bundle/anvien-server.exe` is a launcher wrapper around the same runtime, not a separate backend implementation.

Update the generator or source contract first, then regenerate and verify the generated result.

## Development Setup

Requires Node.js 20+, npm, Git, and Go matching `go.mod` (currently Go 1.26.3).

Install root tooling when you need Prettier, ESLint, Husky, or lint-staged available:

```powershell
npm install
```

Build the Go-backed npm package and CLI:

```powershell
cd anvien
npm install
npm run build
npm link

anvien version
```

For direct source development, run the Go CLI from the repository root:

```powershell
go run ./cmd/anvien version
```

Install the Web UI when changing browser code:

```powershell
cd anvien-web
npm install
```

Package the Windows launcher only when launcher/runtime artifacts changed or when a user-facing packaged build is needed:

```powershell
powershell -ExecutionPolicy Bypass -File anvien-launcher\build.ps1
```

The launcher build uses `ANVIEN_GO` when set, otherwise it falls back to the Go toolchain from `go.mod` and the local `go` command.

## Local Runtime Workflow

Index a repository:

```powershell
cd C:\path\to\repo
anvien analyze .
```

Force a clean rebuild:

```powershell
anvien analyze . --force
```

Configure MCP/editor access:

```powershell
anvien setup
```

Run MCP manually:

```powershell
anvien mcp
```

Run the local Web backend from source:

```powershell
go run ./cmd/anvien serve --host 127.0.0.1 --port 4848
```

Or from the built package:

```powershell
cd anvien
npm run serve
```

Or, after `npm link`:

```powershell
anvien serve
```

Run the Web UI dev server:

```powershell
cd anvien-web
npm run dev
```

The default local endpoints are:

```text
backend: http://127.0.0.1:4848
web:     http://127.0.0.1:5228
```

Use `anvien doctor` when diagnosing local runtime, package, or toolchain issues.

## Validation

Run checks based on what you changed. Do not run the entire suite by habit for a text-only correction, but do verify the source of any documentation that describes code behavior.

Documentation text/formatting only:

```powershell
git diff --check
```

Documentation that describes current codebase behavior:

```powershell
anvien analyze --force
anvien query "<topic>" --repo Anvien
git diff --check
```

Use Anvien and source/tests to understand behavior before changing docs. For the final doc-only commit gate, do not run Anvien change detection; use `git diff --check` and commit the documentation diff.

Root formatting/linting:

```powershell
npm run format:check
npm run lint
```

CLI/core/MCP/backend changes:

```powershell
go test ./cmd/... ./internal/... -count=1
cd anvien
npm run build
npm test
```

Package metadata or runtime packaging changes:

```powershell
cd anvien
npm run build
npm test
npm pack --dry-run --ignore-scripts --json
```

Web UI changes:

```powershell
cd anvien-web
npm run build
npm test
```

Web E2E or browser behavior changes:

```powershell
go run ./cmd/anvien serve --host 127.0.0.1 --port 4848

cd anvien-web
npm run dev
npm run test:e2e
```

Playwright is configured for `http://127.0.0.1:5228`; keep the Web dev server on that port unless you also update the e2e config.

Launcher changes:

```powershell
powershell -ExecutionPolicy Bypass -File anvien-launcher\build.ps1
```

Graph/runtime implementation work must also follow the repository agent rules:

- Refresh graph evidence with `anvien analyze --force` before graph-based analysis.
- Run impact analysis before editing functions, classes, methods, exported symbols, API handlers, graph builders, resolvers, analyzers, or shared contracts.
- Treat HIGH/CRITICAL blast radius as a warning to proceed carefully, not as a ban on editing.
- Run `anvien detect-changes --repo Anvien --scope all` before committing implementation work.

For graph/runtime changes, manually verify repo switching across at least two indexed local repos through the Web UI.

## Hooks And Staged Formatting

The root package defines Husky and lint-staged tooling, but this repository currently does not contain a committed `.husky/` hook directory. Do not rely on hooks as the only verification.

If you add or edit hooks, `.github/`, issue templates, or PR templates, keep paths aligned with the current package names: `cmd/`, `internal/`, `anvien/`, and `anvien-web/`.

Run the validation commands relevant to the area you touched even when local staged formatting succeeds.

## Pull Requests

Use short-lived branches and keep diffs focused. A PR should explain:

- what changed
- why it changed
- how it was verified
- risk, rollback, or migration notes if relevant

PR titles must follow conventional-commit format:

```text
<type>[(scope)][!]: <subject>
```

Allowed types:

| Type | Label |
|------|-------|
| `feat` | `enhancement` |
| `fix` | `bug` |
| `perf` | `performance` |
| `refactor` | `refactor` |
| `docs` | `documentation` |
| `test` | `test` |
| `ci` | `ci` |
| `build` / `deps` | `dependencies` |
| `chore` / `revert` | `chore` |

Use `!` or `BREAKING CHANGE:` in the PR body for breaking changes.

Examples:

```text
feat(web): add repo-scoped graph loading
fix(server): keep graph reads isolated by repo
docs: refresh local runtime runbook
ci: update workflow paths for anvien packages
```

## Coding Guidelines

- Prefer existing package boundaries and local helper APIs.
- Keep write scopes small and avoid unrelated refactors.
- Use structured parsers or existing graph/runtime APIs instead of ad hoc string processing when available.
- Update `README.md`, `RUNBOOK.md`, `ARCHITECTURE.md`, `TESTING.md`, or `CONTRIBUTING.md` when public behavior changes.
- When updating generated AI context behavior, update generator-owned source and tests; generated `.claude/skills/anvien/**`, `AGENTS.md`, and `CLAUDE.md` are validation output.
- Never commit secrets, tokens, real `.env` files, or machine-specific credentials.
- Be careful with generated or repo-local index data under `.anvien/`; it is runtime data, not source code.

## CI And Release Automation

Current workflow intent:

- `.github/workflows/ci.yml` orchestrates quality, tests, and E2E.
- `.github/workflows/ci-quality.yml` owns format, lint, typecheck, Web typecheck, and workflow-convention checks.
- `.github/workflows/ci-tests.yml` owns Go/package/Web test coverage plus Docker config and cross-platform package checks.
- `.github/workflows/ci-e2e.yml` owns Playwright browser flows.
- `.github/workflows/pr-description-check.yml` validates PR descriptions.
- `.github/workflows/pr-labeler.yml` validates PR titles and applies labels.
- `.github/workflows/publish.yml` publishes stable `v*` tags to npm.
- `.github/workflows/release-candidate.yml` publishes RC builds from `master`.
- `.github/workflows/docker.yml` builds tagged Docker images.
- `.github/workflows/tree-sitter-upgrade-readiness.yml`, `.github/workflows/triage-sweep.yml`, and Claude workflows handle repository maintenance and automation.

Workflow conventions:

- Entry-point workflows should have a top-level `concurrency:` block.
- `workflow_call`-only reusable workflows should not own top-level concurrency; the caller should own cancellation.
- `ci.yml` can be called by release workflows and uses a literal `CI-` concurrency prefix to avoid caller/workflow key collisions.
- Use `.github/scripts/check-workflow-concurrency.py` and existing workflow comments as the source of truth when editing CI.

Before relying on release automation, verify package paths and package names still use Anvien names.

## Release Notes

Update `CHANGELOG.md` for user-visible changes. Use `docs/plans/` as supporting history when a change is part of a planned migration or investigation.

Stable releases are tag-driven:

```powershell
git tag vX.Y.Z
git push origin vX.Y.Z
```

Before tagging, make sure these files agree on the version and release notes:

- `internal/version/version.go`
- `anvien/package.json`
- `anvien/package-lock.json`
- `README.md`
- `RUNBOOK.md`
- `CHANGELOG.md`
