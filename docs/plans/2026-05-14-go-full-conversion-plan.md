## Rules

1. Use Anvien for codebase analysis and impact checks while working on this plan.
2. As each task is completed, update the corresponding checklist item immediately.
3. Run a full build before testing; the test suite must include an e2e test.
4. Record benchmark results as each benchmarkable task is completed. Benchmarkable means measured product/runtime performance, capacity, package/startup size, graph/DB throughput, or conversion-inventory counts; build/test/e2e timings are validation evidence unless the slice changes those systems.
5. Record evidence as each evidenced task is completed.
6. commit doc only do not use anvien.
7. After each completed implementation slice, commit the work, then continue until the full plan is complete.
8. Treat generator/scaffold/template-writer files as authority over the files and directories they create.

## Rule Clarifications

### Full Build and Test Suite

The rule "Run a full build before testing; the test suite must include an e2e test" means:

- first run the full build, including Go runtime, launcher, and Web UI build;
- then run the test suite for the completed slice;
- that test suite must include at least one e2e test;
- e2e is one required component of the test suite, not a replacement for focused, integration, or broader regression tests;
- do not treat a single easy Playwright/browser test as sufficient unless it exercises the slice's real end-to-end path.

### E2E Selection

Anvien has two mechanisms:

- Primary mechanism: CLI/MCP runtime for AI/codebase understanding.
- Secondary mechanism: Web UI as a viewer for human inspection of graph/runtime results.

E2E must be selected by the mechanism affected by the slice:

- For CLI, MCP, indexing, graph, query, context, impact, detect-changes, repo storage, setup, package/runtime, or core conversion slices, the e2e must be CLI/MCP/runtime e2e. It should exercise a real repo/index/graph/tool path through the built Go runtime.
- For Web viewer, HTTP bridge, frontend state, graph rendering, repo selection, analyze-from-UI, or session viewer slices, the e2e must be FE-backend e2e through the Go backend and browser UI.
- UI-only/browser tests that only click buttons or check availability are not valid as the plan's required e2e unless the slice itself is strictly UI-only and the test still proves the complete affected behavior.
- Web e2e must not be used as proof for CLI/MCP/setup behavior. CLI/MCP/setup behavior requires CLI/MCP/config tests or smoke evidence.

### Benchmark Meaning

Benchmark ledger entries must be real benchmark measurements:

- runtime/product performance;
- graph/database/index throughput;
- MCP/CLI latency or throughput;
- memory/capacity/package/startup size;
- conversion inventory counts when the task is inventory-measurable.

Build time, unit-test duration, e2e duration, command wall time for validation, and "how long the agent took" are evidence only, not benchmark entries.

### Batch Size

Work must proceed by meaningful clusters:

- convert/delete a full related group of files/tests before build/test;
- do not build/test/commit after each individual file unless the whole cluster is genuinely one file;
- after each cluster passes validation and is committed, continue immediately to the next open checklist item until the plan is complete or a real blocker is recorded.

## Goal

Convert the repository to Go as the implementation authority for every non-Web UI surface.

- Convert 1-1.
- Không refactor lớn.
- Luôn convert test.
- Luôn xóa source cũ sau khi pass.
- Luôn cập nhật ledger.
- Luôn rà soát generator/output ownership trước khi convert hoặc xóa file được sinh tự động.
- Nếu không chắc ownership của file, dừng và ghi blocker và phase jump sang cụm an toàn khác, sau khi xong những cụm liên quan và đủ nội dung kiến thức để convert cụm cũ thì jump về phase chứa cụm đó.

The only TypeScript/React allowed after final cutover is the Web UI display/build surface and generated browser-side glue that is owned by Go contracts.

## Generator And Output Ownership Rule

Some files in this codebase create or update other files/directories automatically. Those generator/scaffold/template-writer files are the real authority; generated output is evidence, not the primary owner.

Before converting, deleting, or classifying any generated-looking file, generated directory, template output, setup output, skill/plugin output, snapshot, baseline, glue file, or packaged config:

- Identify the generator/writer owner first.
- Convert the generator/writer 1-1 before trusting or deleting its output.
- Convert tests around the generator/writer, not only tests around the produced file.
- If generated output changes after converting the generator, record that output change as expected evidence in the ledger.
- If ownership is unclear, stop that file, write the blocker inline in the checklist item, phase jump to a safe cluster, and return only after the generator owner is known.

## Conversion Method

Work in small slices. Each slice must name:

- source files being replaced
- Go target files
- tests being converted
- validation commands
- benchmark entry, if measurable
- evidence entry
- files deleted after the Go replacement passes
- commit hash after completion

Do not start broad redesign during conversion. Accuracy and runtime completeness outrank optimization. Benchmark results are required evidence, but heavy optimization belongs after conversion unless a benchmark proves a correctness or usability blocker.

## Ownership Classification

- Go authority: `cmd/`, `internal/`, Go launcher source.
- Allowed TypeScript: `anvien-web/` UI only, plus generated browser glue owned by Go contracts.
- Must convert/delete: `anvien/src`, `anvien-shared`, non-Web `anvien/test`, non-Web `anvien/scripts`.
- Final cutover decision: `anvien/src`, `anvien-shared`, non-Web `anvien/scripts`, legacy parser/vendor JS glue, and `anvien/tsconfig.json` are deleted or replaced by Go/package lifecycle authority. `anvien-web/` remains the allowed TypeScript/React UI surface. `eslint.config.mjs` remains classified as root Web/dev lint config, not CLI/backend runtime authority.

## Legacy Test Conversion Rule

Legacy non-Web TypeScript/Vitest tests may be converted in bulk only when the corresponding Go implementation already exists.

Rules:

- Convert behavior clusters, not random individual test files.
- Convert in batches large enough to remove meaningful Node/Vitest surface. Do not build/test after each individual file.
- Validation is per completed batch/slice: full build first, then focused Go tests, affected legacy/Web tests if still relevant, at least one e2e, benchmark/evidence, detect changes, commit.
- Place Go tests near the Go package that owns the behavior.
- If Go implementation already exists, convert the TS test cluster now, prove Go coverage, then delete the replaced TS tests.
- If Go implementation does not exist yet, do not write failing/contract Go tests first and do not delete the TS tests. Phase jump to the implementation owner, finish the Go implementation, then return to convert/delete the TS test cluster.
- Keep `anvien/test/fixtures` only when evidence proves the files are source-language fixture input, not runtime or test harness authority.
- Keep `anvien-web` tests when they test the allowed Web UI or generated browser glue.
- Remove `anvien/vitest.config.ts` and non-Web Vitest runner scripts only after all non-Web TS test clusters are converted, deleted, or classified as fixture-only.

## Phase Jump Rule

Every executable work item is a checklist item. If a task must jump to another phase, write the jump directly inside the same checklist item.

Use this format:

```md
- [ ] [P2-A] Convert package X. Phase jump: blocked by ownership uncertainty in Y; jump to [P5-C] to convert safe helper Z, then return to [P2-A].
- [ ] [P5-C] Convert helper Z. Phase jump return: after completion, return to [P2-A].
```

Do not create separate loose notes for phase jumps. The checklist is the ledger.

## Validation Baseline

For implementation slices:

1. Run Anvien graph refresh before graph-based analysis.
2. Run impact analysis before editing any function, class, method, or shared symbol.
3. Run full build before tests.
4. Run focused tests for the slice.
5. Run at least one e2e test.
6. Run benchmark if the task is benchmarkable.
7. Record evidence and benchmark ledgers.
8. Delete replaced non-Web TypeScript/JavaScript source and converted legacy tests after Go tests pass.
9. Run detect changes before commit.
10. Commit the slice.

For docs-only changes:

- Do not use Anvien.
- Run docs-safe validation only, such as `git diff --check`.
- Commit only documentation changes if a commit is requested.
